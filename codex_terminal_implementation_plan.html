Implementation Plan

codex_terminal

Phased engineering plan for building v1 from the approved scope, product spec, and methodology.

1. Objective

This plan turns the approved documents into an execution sequence. The goal is to build a working v1 without architectural drift, premature bells and whistles, or avoidable rework.

2. Build Strategy

Build the app in layers, not pages first. The order should be:

  1. data and metadata foundation,
  2. core analytics engines,
  3. portfolio and comparison logic,
  4. UI assembly and interpretation layer.

This avoids building pretty empty pages before the underlying logic is stable.

3. Phase Breakdown

4. Detailed Work Plan

Phase 1: Project Skeleton And Data Layer

  • Create the Streamlit app structure and navigation shell.
  • Set up package structure for data, analytics, config, and UI modules.
  • Implement cached price-history fetching from yfinance.
  • Implement cached macro-series fetching from FRED.
  • Add common utilities for return calculations, resampling, and missing-data handling.
  • Add visible status reporting for fetch failures and degraded series.

Phase 2: Universe Metadata And Benchmarks

  • Encode the frozen ETF universe and sleeve metadata from the methodology doc.
  • Define sleeve tags: asset class, region, factor, diversifier, inflation sensitivity, duration sensitivity, strategic-core eligibility, tactical-overlay eligibility.
  • Implement permanent benchmark handling for SPY.
  • Implement the Vanguard benchmark fetch attempt and manual fallback workflow.
  • Create benchmark state objects that distinguish between full-history, partial-history, and point-only availability.

Phase 3: Macro And Screener Engines

  • Implement the four-quadrant macro regime model using FRED inputs.
  • Implement tactical signal calculations: absolute momentum, relative momentum, trend, vol-adjusted strength.
  • Implement structural signal placeholders and credible first-pass logic for value and carry where available.
  • Implement block-level score normalization and composite scoring.
  • Implement prescriptive label mapping: overweight, neutral, underweight, avoid, watchlist.
  • Add signal availability and confidence reporting where a sleeve lacks full signal coverage.

Phase 4: Portfolio Lab

  • Implement portfolio return engine.
  • Implement portfolio statistics engine.
  • Implement rolling-return and drawdown analytics.
  • Implement target-volatility logic anchored to SPY.
  • Implement mean-variance optimizer with practical constraints.
  • Implement target-risk-contribution construction.
  • Implement stress-aware scoring overlays using downturn windows and downside metrics.
  • Implement leverage and financing-drag reporting.
  • Implement weight-drift and vol-triggered rebalance logic.

Phase 5: Comparison Engine

  • Implement uploaded portfolio schema validation.
  • Implement direct mapping of known ETFs to sleeve metadata.
  • Implement fallback classification for unknown ETFs and unresolved holdings.
  • Implement comparison tables against SPY, Vanguard benchmark, and Market Beating Portfolio.
  • Implement benchmark-relative factor and sleeve exposure diagnostics.
  • Generate concise regime-aware commentary on exposure differences.

Phase 6: Terminal And Learn Layers

  • Assemble the Terminal page from existing screener, benchmark, and macro outputs.
  • Add the “What changed / What matters / What to watch” summary box.
  • Add the lightweight Learn page with short academic interpretation tied to live outputs.
  • Refine information density and chart hierarchy for institutional usability.

Phase 7: Hardening

  • Handle missing data cleanly throughout the app.
  • Handle partial benchmark histories.
  • Test unstable tickers and degraded fetch states.
  • Validate analytics for obvious numerical errors.
  • Reduce redundant API pulls through caching and shared intermediate data.
  • Clean up layout, copy, and warnings.

6. Minimum Acceptance Criteria By Phase

Phase Done Means
1The app runs locally and can fetch/cache market and macro data for the starter universe.
2The full universe and benchmark stack can be loaded with visible status for degraded benchmarks.
3The Screener and Macro pages produce stable ranks and regime labels.
4The Portfolio Lab can build, backtest, and compare candidate portfolios.
5An uploaded portfolio can be validated and compared to all benchmarks.
6The Terminal page summarizes the system coherently and the Learn page adds concise interpretation.
7Common failure modes are visible, analytics are credible, and the app is usable end to end.

7. Key Risks To Manage During Build

  • Scope creep from trying to support too many ETFs too early.
  • Overfitted screener logic before the first honest baseline exists.
  • Weak value or carry proxies being treated as high-confidence signals.
  • Benchmark fragility around Vanguard data access.
  • UI bloat before analytics are stable.

8. Build Recommendation

The correct move now is to start implementation with Phases 1 and 2 together. Those phases create the foundation for every other module and will flush out data and universe issues early.

Recommended start: begin coding the app skeleton, data fetch layer, universe config, and benchmark plumbing first. Do not start with page styling or optimization UI.
