V1 Product Specification

codex_terminal

Implementation-facing specification for the first working version of the cross-asset research and portfolio terminal.

1. Purpose Of This Document

This document translates the approved project scope into a buildable v1 specification. It defines what must be in the first version, what can be simplified, what is explicitly out of scope, and what design rules should govern implementation.

2. V1 Goal

Build a Streamlit application that lets the user:

  • monitor cross-asset market leadership and deterioration,
  • evaluate broad and sub-bucket asset-class signals,
  • design and test a house “Market Beating Portfolio,”
  • compare any uploaded portfolio against SPY, a Vanguard target retirement benchmark, and the house portfolio,
  • interpret all of the above through a rules-based macro regime framework.

3. V1 Success Criteria

  • The app produces a credible cross-asset daily screener using free public data.
  • The app can backtest and optimize ETF-based portfolios with benchmark-relative outputs.
  • The app can explain differences between an uploaded portfolio and the benchmark stack.
  • The app can summarize the current macro regime using FRED-backed rules.
  • The app remains opinionated and institutionally framed rather than endlessly customizable.

4. Required Pages / Modules

Page Required In V1 Main User Question
Terminal Yes What matters in markets right now?
Screener Yes What asset sleeves are strong, weak, expensive, attractive, or regime-favored?
Portfolio Lab Yes How should the Market Beating Portfolio be constructed and tested?
Compare Yes How does this input portfolio differ from the benchmarks?
Macro Yes Which macro regime is in force and what should that imply?
Learn Yes, light version What am I missing about today’s market and portfolio tradeoffs?

5. Data Sources

5.1 Mandatory Sources

  • Yahoo Finance / yfinance for ETF market data and benchmark history where available.
  • FRED for macro regime inputs.

5.2 Conditional Sources

  • AQR capital market expectation source for display-only reference data.
  • Vanguard target retirement data source if free, stable access is feasible.

5.3 Data Rules

  • Free public data only in v1.
  • No paid feed dependencies.
  • No source sprawl. Any new source beyond the above requires approval.
  • Daily data is sufficient for the screener and terminal in v1.
  • Monthly resampling is acceptable for some long-horizon analytics and factor regressions where appropriate.

7. Initial V1 Asset Coverage

V1 should cover a practical starter universe rather than every possible ETF. The starter universe should be broad enough to represent the core thesis.

7.1 Coverage Buckets

  • US equity beta
  • US small cap
  • US value
  • US momentum proxy where credible
  • International developed equity
  • International value / small cap where feasible
  • Emerging markets
  • Short, intermediate, and long Treasuries
  • Inflation-linked bonds
  • Investment-grade or broad bond proxy
  • REITs
  • Broad commodities
  • Gold
  • Managed-futures proxy if implementation quality is acceptable
  • Cash or ultra-short Treasury proxy

7.2 Universe Freeze Requirement

Before coding begins, the first build should lock a named list of tickers for each sleeve. That separate instrument list is a dependency of implementation and should be treated as a companion spec.

8. Benchmark Stack

8.1 Always-On Benchmarks

  • SPY
  • Market Beating Portfolio
  • Vanguard target retirement benchmark, if available

8.2 Vanguard Benchmark Fallback Logic

  1. Attempt automated fetch of actual NAV/history.
  2. If unavailable, show benchmark status as degraded.
  3. Prompt user for the latest close or NAV manually.
  4. Use the manually entered figure for current benchmark display and note limitations.

9. Terminal Page Specification

9.1 Purpose

Provide a dense but interpretable snapshot of the market environment.

9.2 Required Components

  • Current macro regime badge
  • Top-performing and worst-performing major sleeves over selectable horizons
  • Market breadth or participation summary if feasible from the chosen universe
  • Factor leadership snapshot
  • Market Beating Portfolio current summary
  • Short commentary box: “What changed / What matters / What to watch”

9.3 Time Horizons

Minimum required selectable lookbacks: 1 month, 3 months, 6 months, 12 months, and year-to-date.

10. Screener Specification

10.1 Required Scope

The screener must work at two levels:

  • Broad asset-class level
  • Sub-bucket or sleeve level

10.2 Required Signal Columns

  • Absolute momentum
  • Relative momentum
  • Trend
  • Volatility-adjusted strength
  • Value proxy or valuation context where feasible
  • Carry proxy where feasible
  • Macro regime fit
  • Composite score
  • Prescriptive stance

10.3 V1 Signal Methodology Rules

  • Use simple, explainable formulas first.
  • Avoid fragile or overly parameterized signal engineering in v1.
  • If a signal cannot be built credibly for a sleeve, mark it unavailable rather than fabricate precision.
  • The composite should remain interpretable by exposing underlying sleeve contributions.

10.4 Composite Design

V1 composite should follow a structured blend:

  • Tactical block: trend, absolute momentum, relative momentum, volatility-adjusted strength
  • Structural block: value, carry, medium-term expected return context
  • Macro block: regime fit used as context or mild influence

Implementation note: expose block-level sub-scores as well as a final score.

10.5 Prescriptive Mapping

The screener should map final score bands to:

  • Overweight
  • Neutral
  • Underweight
  • Avoid
  • Watchlist

These mappings should be configurable internally in code but not presented as a heavy user customization feature.

11. Macro Module Specification

11.1 Framework

Use a four-quadrant rules-based regime model:

  • Growth up / inflation up
  • Growth up / inflation down
  • Growth down / inflation up
  • Growth down / inflation down

11.2 V1 Output Requirements

  • Current regime classification
  • Regime trend or transition signal if detectable
  • Underlying macro inputs used in classification
  • Short explanation of what assets and factors tend to work or fail in this regime

11.3 Regime Use Across App

  • Display on Terminal page
  • Use in Screener as context or mild scoring influence
  • Use in Compare commentary
  • Use in Learn page explanations

12. Portfolio Lab Specification

12.1 Core Purpose

Construct and test the Market Beating Portfolio as a strategic core plus tactical overlay that targets SPY realized volatility and seeks superior long-term risk-adjusted outcomes.

12.2 Required Portfolio Construction Modes

  • Manual-weight portfolio builder
  • Target-volatility portfolio builder
  • Target-risk-contribution portfolio builder
  • Mean-variance optimizer

12.3 Required Output Panels

  • Weights table
  • Risk contribution table
  • Performance statistics table
  • Rolling return charts
  • Drawdown chart
  • Correlation matrix or correlation diagnostics
  • Benchmark-relative comparison vs SPY
  • Stress period performance table
  • Leverage and financing-drag summary

12.4 Rebalancing Rule

Default to volatility-triggered or weight-drift-triggered rebalancing. Calendar-only rebalancing should not be the default frame in v1.

12.5 Tail-Risk Logic

V1 optimization should be stress-aware. Candidate portfolios should be evaluated not only by return and volatility, but also by:

  • maximum drawdown,
  • downside deviation,
  • crisis-period relative resilience,
  • correlation breakdown in stress,
  • behavior in inflation and growth shock periods.

12.6 Leverage Rule

Default leverage implementation should be volatility targeting to SPY realized volatility with a visible financing assumption. No brokerage-specific margin engine is required in v1.

12.7 Concentration Rule

Soft limits only. If a position exceeds a house hard-limit threshold, the UI should flag it prominently but not block the portfolio.

13. Comparison Module Specification

13.1 Input

At minimum, the app should accept an uploaded portfolio file containing ticker and weight columns. Shares or notional can be supported later if cleanly derivable.

13.2 Required Validation

  • Check for duplicate tickers
  • Check for missing weights
  • Check for weights summing materially above or below 100 percent
  • Flag unknown or unsupported tickers

13.3 Required Analysis

  • Performance vs SPY, Vanguard benchmark, and Market Beating Portfolio
  • Risk vs those same benchmarks
  • Asset-class exposure differences
  • Factor exposure differences
  • Qualitative summary of what risks the uploaded portfolio is more and less exposed to
  • Current-regime commentary on how those differences matter now

13.4 Exposure Inference Rule

V1 should infer exposures from holdings as far as possible using available ticker-level mappings and proxy logic. If a holding cannot be reliably classified, the app should surface that uncertainty rather than hide it.

14. Learn Module Specification

This should remain lightweight in v1. It is not a textbook module.

  • Explain key terms behind today’s outputs
  • Highlight what the user may not know about current market conditions
  • Connect regime, factors, and benchmark behavior to concise academic framing
  • Keep commentary practical and short

15. Required Analytics

  • CAGR
  • Annualized volatility
  • Sharpe ratio
  • Sortino ratio
  • Maximum drawdown
  • Rolling returns
  • Correlation diagnostics
  • Factor regression / factor attribution
  • Efficient frontier or equivalent optimizer chart
  • Rebalancing analysis
  • Regime-based performance slicing

16. Performance And Resource Constraints

  • Do not optimize for instant response if it compromises analysis quality.
  • Backtests taking up to about a minute are acceptable.
  • Memory use should be disciplined, but broad free-data usage is acceptable.
  • Prefer caching and reusable intermediate data rather than excessive repeated API pulls.

17. UX Requirements

  • High-density layout is acceptable.
  • Separate pages or modules are preferred over one overloaded dashboard.
  • Outputs should feel institutional rather than retail.
  • Explanations should be direct and useful, not verbose.
  • The app should remain opinionated for the house framework.

18. Strict V1 Non-Goals

  • No broker integration
  • No trade execution
  • No single-stock research module
  • No tax analysis
  • No WSJ or Bloomberg news ingestion in v1
  • No embedded LLM assistant in v1
  • No attempt at full Bloomberg-terminal command behavior

19. Open Dependencies Before Coding

  1. Freeze the initial ETF ticker universe for each sleeve.
  2. Define exact signal formulas and lookbacks for the screener.
  3. Define exact macro inputs and regime-threshold rules.
  4. Define benchmark file or fetch logic for Vanguard target retirement.
  5. Define the upload schema for the Compare module.
Implementation posture: coding should not begin until the v1 universe and signal formulas are frozen. The app is conceptually clear now, but those two pieces determine whether the first build is coherent or noisy.

20. Recommended Build Order

  1. Data layer and cached fetch infrastructure
  2. Universe definition and benchmark plumbing
  3. Macro regime engine
  4. Screener engine
  5. Portfolio Lab engine
  6. Comparison engine
  7. Terminal summary layer
  8. Learn-page interpretation layer
